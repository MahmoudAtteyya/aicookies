🚀 Quick Start

Fully OpenAI-compatible. Use the standard endpoint + your API key + any model from the list below.

POST https://aicookies.elliaa.com/v1/chat/completions

🔐 Authentication

All requests require a Bearer token:

Authorization: Bearer YOUR_AICOOKIES_API_KEY

cURL

curl -X POST https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_AICOOKIES_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"mistral-small","messages":[{"role":"user","content":"Hello!"}]}'

Python (OpenAI SDK)

from openai import OpenAI
client = OpenAI(
    base_url="https://aicookies.elliaa.com/v1",
    api_key="YOUR_AICOOKIES_API_KEY"
)
response = client.chat.completions.create(
    model="mistral-small",
    messages=[{"role":"user","content":"Hello!"}])
print(response.choices[0].message.content)

Node.js

const res = await fetch("https://aicookies.elliaa.com/v1/chat/completions", {
  method:"POST",
  headers: {
    "Content-Type":"application/json",
    "Authorization":"Bearer YOUR_AICOOKIES_API_KEY"
  },
  body: JSON.stringify({model:"mistral-small", messages:[{role:"user",content:"Hello!"}]})
});

Streaming (SSE)

Add "stream": true to the request body for real-time token-by-token streaming:

curl -X POST https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_AICOOKIES_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"mistral-small","stream":true,"messages":[{"role":"user","content":"Hello!"}]}'

🍪 Claude Response (Cookie-based)

Claude uses browser cookies instead of API keys. The gateway includes a natural prompt middleware that converts OpenAI-format API requests into natural conversation prompts — preserving system instructions, multi-turn context, and all message details without any external API calls.

# Non-streaming response — standard OpenAI format
{
  "model": "claude-sonnet-4-6",
  "choices": [{"message": {"content": "مرحباً بك! 😊"}, "finish_reason": "stop"}],
  "usage": {"prompt_tokens": 12, "completion_tokens": 8, "total_tokens": 20}
}

# Streaming response — OpenAI-compatible SSE
data: {"choices":[{"delta":{"content":" مرحباً"}}]}
data: {"choices":[{"delta":{"content":" بك! كيف حالك؟"}}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
data: [DONE]

🔧 Natural Prompt Middleware (replaces Mistral humanization)

The gateway converts API requests into natural conversation without external calls:

  • System prompts → included as natural context at the top (no <system> tags)
  • Multi-turn conversations → full context preserved (all user + assistant messages)
  • No API artifacts → no JSON structures, no role: markers, no formatting that reveals automation
  • Zero latency overhead → pure local transformation (previously added 200-800ms via Mistral API call)
  • Meaning preservation → exact text passed through, no rewriting or paraphrasing

🛡️ Error Handling

All errors return clean JSON (never HTML):

// 400 — Bad Request (invalid JSON, missing messages, last message not from user)
{"error": "Bad Request", "message": "..."}

// 401 — Unauthorized (invalid API key)
{"error": "Unauthorized", "message": "Invalid or missing API key"}

// 503 — All sessions busy (all cookies in cooldown or dead)
{"error": "Service Unavailable", "message": "All Claude sessions are currently unavailable.",
 "tried": [{"token": 7, "action": "COOLDOWN", "reason": "Rate limited"}],
 "retry_after_ms": 300000}

// 504 — Gateway Timeout (Claude didn't respond in 120s)
{"error": "Gateway Timeout", "message": "Provider did not respond in time"}

🧹 Auto-Cleanup & Natural Conversation Naming

After each successful request, the gateway automatically deletes the conversation from Claude.ai to prevent accumulation. Conversations are created with an empty name — Claude auto-names them based on the first message content, just like a normal user conversation. This ensures the sessions look natural and don't trigger spam detection.

🔧 Tool Artifact Filtering (Anti-Leak Protection)

Claude.ai's web interface has tools (web search, MCP, etc.) enabled in its system prompt. When the model decides to use a tool, it emits XML tags like <function_calls> and <invoke name="..."> as plain text, then stops generation to wait for a tool result. Since our gateway doesn't execute tools, these artifacts would leak to the API consumer as raw text.

The gateway handles this in two layers:

  • Layer 1 — Automatic Retry: If Claude's response contains tool artifacts or stops with tool_use reason, the gateway automatically retries with an anti-tool instruction appended to the prompt. This usually makes Claude answer directly.
  • Layer 2 — Regex Filtering: Any remaining tool XML is stripped from the response before sending to the client. Patterns filtered: <function_calls>, <invoke>, <function_results>, <antml:function_calls>, and all variants.

This means API consumers always receive clean text responses — never tool call XML or incomplete responses.

📡 Claude Streaming Format

Claude responses are streamed in OpenAI-compatible SSE format. The gateway transforms Anthropic's native SSE format in real-time:

# Claude SSE (Anthropic format) → OpenAI SSE format
data: {"choices":[{"delta":{"content":"Hello"}}]}
data: {"choices":[{"delta":{"content":" world"}}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
data: [DONE]

Tool artifacts are filtered from each streaming chunk — chunks that are entirely tool XML are silently dropped. The finish_reason mapping:

  • stop_sequence / end_turn → stop
  • max_tokens → length
  • tool_use → tool_calls (triggers automatic retry)

Rate limits: Claude uses session-based limits (per 5 hours / 7 days). The proxy automatically rotates between uploaded cookie sets when one hits the limit. Cloudflare challenges trigger a cooldown (not permanent death) — the token auto-recovers after 5 minutes.

🔄 Smart Key Rotation

  1. Request → least-used active key → real provider API
  2. 429/402 → rotate to next key (free providers recover; Fireworks dies permanently)
  3. All keys busy → returns 503 Service Unavailable with retry_after_ms: 5000
  4. Response includes X-Proxy-Provider, X-Proxy-Key, X-Proxy-Latency headers
