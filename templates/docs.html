1. 🏗️ Project Overview & Architecture

AI Proxy Gateway (AICookies) is a self-hosted, OpenAI-compatible API gateway that aggregates multiple AI providers behind a single endpoint. It proxies requests to 5 providers (Claude, Mistral, Cohere, SambaNova, Fireworks) with automatic key rotation, TLS fingerprinting, conversation affinity, and enterprise-grade reliability features.

3. 🔐 Authentication & API Keys

All API requests require a Bearer token in the Authorization header:

Authorization: Bearer sk-aic-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

API Key Format

  • Prefix: sk-aic- (generated keys) or legacy keys set via PROXY_API_KEY_HASH
  • Generation: secrets.token_urlsafe(32) → 43 chars of cryptographic randomness
  • Storage: SHA-256 hash only (plaintext never stored)
  • Verification: hmac.compare_digest() — constant-time comparison (prevents timing attacks)
Multiple API Keys: The gateway supports unlimited API keys. Generate new keys from the admin panel → /generate-key. Each key works with all 15 models.

Generating a New API Key

# Via admin panel (browser)

# Via API (requires admin session)
curl -X POST https://aicookies.elliaa.com/generate-key \
  -H "Cookie: session=YOUR_ADMIN_SESSION" \
  -d "label=my-app-key"

# Response:
# {"api_key": "sk-aic-a1b2c3d4e5f6...", "key_hash": "e5f6..."}

    {"role": "user", "content": "Hello!"},

8. 📡 Streaming (SSE)

Add "stream": true for real-time token-by-token streaming:

curl -X POST https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_AICOOKIES_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"claude-sonnet-4-6","stream":true,"messages":[{"role":"user","content":"Hello!"}]}'

# SSE Response:
data: {"choices":[{"delta":{"content":" مرحباً"}}]}
data: {"choices":[{"delta":{"content":" بك! كيف"}}]}
data: {"choices":[{"delta":{"content":" حالك؟"}}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
data: [DONE]
SSE Format Details: Claude uses two internal formats — legacy {"completion":"text"} and content_block_delta events. The gateway handles both transparently. API consumers always receive standard OpenAI-compatible chunks.

Streaming + Artifacts

The streaming transformer buffers the full response, applies antArtifact → Markdown transformation and tool XML filtering, then re-chunks into SSE events. Chunks that would be entirely tool XML are silently dropped.

9. 🛠️ Function Calling (Tools) v3.0

The gateway supports OpenAI-compatible tools and tool_choice parameters. When you send function definitions, the gateway injects them as a natural preamble — Claude knows what tools are available and calls them using <function_calls> XML format. The gateway parses the XML and returns standard OpenAI tool_calls in the response.

curl -X POST https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_AICOOKIES_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-sonnet-4-6",
    "messages": [{"role": "user", "content": "What is the weather in Cairo?"}],
    "tools": [{
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get current weather for a city",
        "parameters": {
          "type": "object",
          "properties": {"city": {"type": "string", "description": "City name"}},
          "required": ["city"]
        }
      }
    }],
    "tool_choice": "auto"
  }'

# Response — standard OpenAI tool_calls format
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "",
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {"name": "get_weather", "arguments": "{\"city\": \"Cairo\"}"}
      }]
    },
    "finish_reason": "tool_calls"
  }]
}

Multi-Turn Tool Calls

After executing the tool, send the result back with role: "tool":

{"messages": [
  {"role": "user", "content": "Weather in Cairo?"},
  {"role": "assistant", "content": null, "tool_calls": [{
    "id": "call_abc", "type": "function",
    "function": {"name": "get_weather", "arguments": "{\"city\": \"Cairo\"}"}
  }]},
  {"role": "tool", "tool_call_id": "call_abc", "content": "Sunny, 32°C"},
  {"role": "user", "content": "What about tomorrow?"}
]}

11. 🍪 Claude Cookie Engine

Claude models use browser session cookies instead of API keys. The gateway manages the entire lifecycle:

How It Works

  1. Upload: Admin uploads a Netscape cookie file via /upload
  2. Parse: Gateway extracts sessionKey, lastActiveOrg, anthropic-device-id, and Cloudflare tokens
  3. Encrypt: If COOKIE_ENCRYPTION_KEY is set, cookies are encrypted at rest with Fernet
  4. Request: When a Claude request arrives, gateway selects a cookie set (affinity-aware) and sends to claude.ai/api/organizations/{org}/chat_conversations/{conv}/completion
  5. Transform: Gateway converts OpenAI messages → natural conversation prompt, sends via curl_cffi with impersonate=chrome131 to bypass Cloudflare
  6. Stream: Anthropic SSE → OpenAI SSE format in real-time
  7. Cleanup: Conversation is deleted from Claude.ai after each request

Natural Prompt Middleware

The gateway converts API requests into natural conversation without external calls:

  • System prompts → included as natural context at the top (no <system> tags)
  • Multi-turn conversations → full context preserved (all user + assistant messages)
  • No API artifacts → no JSON structures, no role: markers, no formatting that reveals automation
  • Zero latency overhead → pure local transformation
  • Meaning preservation → exact text passed through, no rewriting or paraphrasing

Tool Artifact Filtering (Anti-Leak Protection)

Claude.ai's web interface has tools (web search, MCP) enabled. When the model decides to use a tool, it emits XML tags as plain text. Since our gateway doesn't execute these internal tools, artifacts would leak to the API consumer.

The gateway handles this in two layers:

  • Layer 1 — Automatic Retry: If Claude's response contains tool artifacts or stops with tool_use reason, the gateway retries with an anti-tool instruction appended. This usually makes Claude answer directly.
  • Layer 2 — Regex Filtering: Any remaining tool XML is stripped from the response. Patterns: <function_calls>, <invoke>, <function_results>, <antml:function_calls>, <web_search> tags, and all variants.
Filter safety: The regex patterns match only Claude's internal tool XML tags. They do NOT affect legitimate HTML/SVG/JSX code that Claude writes as part of the answer. Tags like <div>, <svg>, <script> pass through untouched.

Artifact Compatibility (Standard Markdown Output)

Claude.ai uses proprietary <antArtifact> XML tags. The gateway converts them to standard Markdown code blocks:

text/html              → ```html
application/vnd.ant.react → ```tsx
text/python            → ```python
text/javascript        → ```javascript
image/svg+xml          → ```svg
application/json       → ```json
text/markdown          → ```markdown
text/sql               → ```sql
text/yaml              → ```yaml
...and more

Also strips <antThinking> reasoning blocks — users see only the final answer.

Web Search (Native, Server-Side)

The gateway enables Claude's native web search. When a user asks about current events, prices, news, or time-sensitive information, Claude automatically searches the web and returns factual, up-to-date answers — no external API required.

curl -X POST https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer YOUR_AICOOKIES_API_KEY" \
  -d '{"model":"claude-sonnet-4-6","messages":[{"role":"user","content":"What is the current price of gold?"}]}'

# Response — Claude searched the web and returned real data
{"choices":[{"message":{"content":"The current price of gold is $4,085 per troy ounce..."}}]}

TLS Fingerprint Impersonation

curl_cffi + Chrome 131: The gateway uses curl_cffi with impersonate="chrome131" for all Claude API calls. This replicates Chrome's TLS fingerprint (JA3/JA4), HTTP/2 settings, and header ordering — making the traffic indistinguishable from a real Chrome browser. This is critical for bypassing Cloudflare's bot detection.

Rate Limits & Cooldown

Claude uses session-based limits (per 5 hours / 7 days). The proxy automatically rotates between uploaded cookie sets when one hits the limit. Cloudflare challenges trigger a cooldown (not permanent death) — the token auto-recovers after 5 minutes.

12. 🔒 Security Features v3.0

Fernet Cookie Encryption

Uploaded Claude cookies are encrypted at rest using Fernet symmetric encryption (AES-128-CBC + HMAC-SHA256). Set COOKIE_ENCRYPTION_KEY environment variable to enable. If not set, gracefully falls back to plaintext storage.

# Generate a Fernet key for your deployment
python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
# Set in docker-compose.yml:
COOKIE_ENCRYPTION_KEY=your-generated-key-here

API Key Minting

  • Prefix: sk-aic- + 43 chars of secrets.token_urlsafe(32)
  • Storage: SHA-256 hash only (plaintext never persisted)
  • Verification: hmac.compare_digest() — constant-time comparison (prevents timing attacks)
  • Multiple keys: Unlimited keys supported — generate from admin panel

Additional Security Measures

  • Cookie name validation — only safe characters ([\w\-\.]+) accepted in cookie names
  • XSRF auto-extraction — XSRF-TOKEN extracted from cookies and set as x-xsrf-token header automatically
  • Admin session auth — dashboard protected by werkzeug.security password hashing
  • Proxy credentials — IPRoyal proxy credentials stored in environment variables, not in code

13. ♻️ Reliability Features v3.0

Retry Middleware

Failed requests to Claude automatically retry with exponential backoff + jitter. Respects Retry-After headers. Max delay: 60 seconds.

Attempt 1: ~2s delay (if failed)
Attempt 2: ~4s delay + jitter
Attempt 3: ~8s delay + jitter
→ Max 3 attempts, then return error to client

Rate Limiting (Token Bucket)

Per-client-IP rate limiting using a token bucket algorithm (60 requests/minute with burst capacity).

// 429 — Rate limit exceeded
{"error": "Rate limit exceeded", "retry_after_seconds": 15}
// Headers: X-RateLimit-Remaining: 0, Retry-After: 15

Proxy Blacklist

Proxy IPs that fail repeatedly are auto-blacklisted for 5 minutes after 3 consecutive failures. Subsequent requests skip blacklisted proxies automatically.

# Check proxy blacklist status
curl https://aicookies.elliaa.com/v1/health
# → "proxy_blacklist": {"blacklisted_count": 0, "entries": []}

14. 🧠 Orchestration Features v3.0

Conversation Affinity

Multi-turn conversations are pinned to the same Claude cookie using a SHA-256 fingerprint of the first user message. This preserves conversation context across turns.

# Turn 1 → cookie #5 is selected → affinity pin set
{"messages": [{"role": "user", "content": "What is Python?"}]}
# X-Proxy-Rotation: token 5 (used 1x)

# Turn 2 → SAME cookie #5 is used (affinity pin)
{"messages": [
  {"role": "user", "content": "What is Python?"},
  {"role": "assistant", "content": "Python is..."},
  {"role": "user", "content": "Show me an example"}
]}
# X-Proxy-Rotation: token 5 (used 2x) ← same cookie!

Session Tier System

  • Max tier — highest priority, most capacity
  • Pro tier — medium priority
  • Free tier — lowest priority, rate-limited

Tokens are sorted: affinity-pinned first → highest tier → least-used.

15. 💬 Conversation Features v3.0

Title Generation

After each non-streaming Claude response, the gateway generates a conversation title via Claude's title API. The title is included in the response as a title field when available.

{"id": "claude-a1b2c3d4", "title": "Python Programming Basics", "choices": [...]}

Response Caching

Identical requests (same affinity key + prompt hash) are cached for 5 minutes. Cached responses include "cached": true and the X-Cache-Hit: true header — zero latency, zero Claude API calls.

# First request — hits Claude, caches response
# X-Cache-Hit: false

# Identical request within 5 min — served from cache
# X-Cache-Hit: true
# {"cached": true, "choices": [...]}

16. 🔄 Smart Key Rotation

  1. Request → least-used active key → real provider API
  2. 429/402 → rotate to next key (free providers recover; Fireworks dies permanently)
  3. 10 errors → key auto-paused
  4. All keys busy → returns 503 Service Unavailable with retry_after_ms: 5000
  5. Response includes X-Proxy-Provider, X-Proxy-Key, X-Proxy-Latency headers

Token Management Flow

  1. Admin logs in → /login with username + password
  2. Navigates to /generate-key
  3. Clicks "Generate" → new sk-aic-... key created
  4. Key is shown once → admin saves it securely
  5. Key works immediately → usable with all 15 models
  6. Key hash stored in DB → proxy_api_keys table (SHA-256 only)
Important: Generated API keys are shown only once. There is no way to retrieve the plaintext key after generation — only the hash is stored. If lost, generate a new one.

Fireworks AI Failover — How It Works

Request → /v1/fireworks-deepseek/chat/completions
  │
  ▼
proxy_to_provider("fireworks", ...)
  │
  ├─ get_provider_keys("fireworks") → [key1, key2, key3, ...]
  │   (ordered by usage_count ASC — least used first)
  │
  ├─ Try key1:
  │   ├─ HTTP 200 → return response ✅
  │   ├─ HTTP 412 (suspended) → mark_key_dead(key1) → continue to key2
  │   ├─ HTTP 402 (depleted) → mark_key_dead(key1) → continue to key2
  │   ├─ HTTP 429 (rate limit) → mark_key_rate_limited(key1) → continue to key2
  │   └─ HTTP 401/403 (auth) → mark_key_dead(key1) → continue to key2
  │
  ├─ Try key2:
  │   ├─ HTTP 200 → return response ✅
  │   └─ ... same logic
  │
  ├─ Try key3: ...
  │
  └─ All keys failed → 503 with failover details
      {
        "error": "Service Unavailable",
        "tried_keys": 3,
        "errors": ["ACCOUNT_SUSPENDED", "RATE_LIMITED", "ACCOUNT_SUSPENDED"]
      }

Dead Key Recovery: Dead keys are never auto-revived. Admin must click "🔄 Revive" in the dashboard or keys page. The system sends a lightweight GET /models request to check if the account is active again. If 200 → key is restored to active.
