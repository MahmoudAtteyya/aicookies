01

Authentication

#

Every request to the gateway must include a valid API key passed as a Bearer token in the Authorization header. Keys are prefixed with sk- and are issued per-account.

ⓘ
Header format: Authorization: Bearer sk-xxxxxxxxxxxxxxxx

Getting a Key

  • Request a key from the gateway administrator or via the /api/keys management endpoint.
  • Keys are scoped to a provider pool and rotated automatically when rate-limited.
  • Treat your key like a password — never expose it in client-side code.
bash
# Correct — Bearer token in header
curl https://aicookies.elliaa.com/v1/chat/completions \
  -H "Authorization: Bearer sk-your-key" \
  -H "Content-Type: application/json" \
  -d '{"model":"claude-4-opus","messages":[{"role":"user","content":"Hi"}]}'

# ❌ Wrong — key in body or query string will be rejected
# curl ...?api_key=sk-xxx        # 401 Unauthorized
⚠
Missing or malformed Authorization headers return 401 Unauthorized. Keys with invalid characters or incorrect prefix are rejected before any upstream call is made.

Streaming (SSE)

#

Set "stream": true to receive tokens as Server-Sent Events as they are generated. Each event is a data: line containing a JSON chunk; the stream terminates with data: [DONE].

POST /v1/chat/completions stream: true

Request

json
{
  "model": "claude-4-opus",
  "messages": [{ "role": "user", "content": "Count to 5 slowly." }],
  "stream": true
}

SSE Response

text/event-stream
data: {"id":"chatcmpl-1","choices":[{"delta":{"role":"assistant"}}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{"content":"1"}}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{"content":", 2"}}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{"content":", 3"},"finish_reason":null}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{},"finish_reason":"stop"}]}

data: [DONE]
ⓘ
Consuming SSE: Parse lines prefixed with data: , strip the prefix, JSON-decode the remainder. Stop when you receive the literal string [DONE]. Empty lines are keep-alive separators.

Function Calling

#

Pass a tools array of function definitions and the model can choose to emit a structured tool_calls response instead of (or alongside) text. Supported on all models — the gateway normalizes tool-call schemas across providers.

Tool Definition Format

json
{
  "model": "gpt-5",
  "messages": [
    { "role": "user", "content": "What's the weather in Tokyo?" }
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": {
          "type": "object",
          "properties": {
            "city": { "type": "string", "description": "City name" }
          },
          "required": ["city"]
        }
      }
    }
  ],
  "tool_choice": "auto"
}

tool_choice Values

ValueBehavior
"auto"Model decides whether to call a tool. Default.
"none"Never call tools; force a text response.
"required"Force the model to call at least one tool.
{"type":"function","function":{"name":"get_weather"}}Force a specific function.

Response with tool_calls

{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": null,
        "tool_calls": [
          {
            "id": "call_abc123",
            "type": "function",
            "function": {
              "name": "get_weather",
              "arguments": "{\"city\":\"Tokyo\"}"

          }

Native Web Search

#

Some models (notably xAI Grok and select Anthropic models) support a built-in web search tool. Enable it by passing a web_search_v0 entry in the tools array — no function schema required. The gateway passes the request to the provider's native search backend.

ⓘ
Supported on: Models whose provider exposes a native search tool. If the upstream provider doesn't support search, the tool is silently ignored and the model answers from its training data.

Claude Cookie Proxy

#

For Claude models, the gateway can proxy requests through the claude.ai web session using authenticated cookies rather than the official API. This unlocks Claude access when API keys are exhausted or unavailable. Requests are sent using curl_cffi with a spoofed TLS fingerprint to bypass bot detection.

⚠
Cookies are session-scoped and expire. The gateway's key rotation automatically refreshes and falls back between API keys and cookie pools — you don't need to manage this manually.

How It Works

  1. The gateway holds a pool of Claude session cookies (per-account sessionKey from claude.ai).
  2. When a Claude model is requested, the gateway builds a web-style request body.
  3. The request is sent via curl_cffi using the Chrome 120 TLS JA3 fingerprint and matching HTTP/2 headers.
  4. The streaming response is parsed and re-emitted in OpenAI-compatible SSE chunks.

Under the hood

python
from curl_cffi import requests

# TLS fingerprint impersonation — looks like a real Chrome browser
resp = requests.post(
    "https://claude.ai/api/organizations/{}/chat_conversations",
    json=payload,
    cookies={"sessionKey": cookie},
    impersonate="chrome120",   # JA3 + HTTP/2 fingerprint
    headers={
        "User-Agent": "Mozilla/5.0 ... Chrome/120.0.0.0 ...",
        "anthropic-client-sha": "...",
    },
    stream=True,
)
!
Best-effort fallback. Cookie-based proxying is not guaranteed — claude.ai may rate-limit or invalidate sessions. The gateway treats cookie 429/403 responses the same as key failures and rotates to the next available credential.

Smart Key Rotation

#

The gateway maintains a pool of API keys (and Claude cookies) per provider. On each request it selects the best-available key using a state machine that tracks health, cooldown timers, and dead-key quarantines.

Rotation Rules

Upstream StatusActionKey StateCooldown
200 OKContinue using keyhealthy—
429 Too Many RequestsRotate to next key; mark current as coolingcooldown60s
401 / 403Mark key as dead; rotate immediatelydeadpermanent
500 / 502 / 503Retry once on same key, then rotatecooldown30s
504 TimeoutRotate to next key immediatelycooldown15s
Network errorRotate; mark key as unstablecooldown20s
Cookie 403 (Claude)Drop cookie; pick fresh onedeadpermanent

Selection Algorithm

pseudo
# Pseudocode — key selection per provider
def pick_key(provider):
    keys = pool[provider].filter(state="healthy")
    if keys:
        return round_robin(keys)      # load balance

    # No healthy keys — try any whose cooldown has elapsed
    cooled = pool[provider].filter(cooldown_elapsed=True)
    if cooled:
        return round_robin(cooled)

    raise AllKeysExhausted("503 no available keys for {provider}")

Error Codes

#

All errors are returned as JSON in the OpenAI error envelope: {"error":{"message":...,"type":...,"code":...}}.

HTTPTypeMessageCause
400invalid_request_errorMalformed JSON or missing required field.Bad request body.
401authentication_errorMissing or invalid API key.No/invalid Bearer token.
403permission_errorKey lacks access to this model/provider.Key not provisioned for model.
404not_found_errorUnknown model slug or endpoint.Invalid model or path.
405method_not_allowedHTTP method not supported on this path.e.g. GET on a POST-only route.
429rate_limit_errorRate limit exceeded. Retry after Ns.Token bucket empty.
500internal_errorInternal gateway error.Unexpected server fault.
502upstream_errorUpstream provider returned an error.Bad gateway response.
503service_unavailableNo available keys for the requested provider.All keys in cooldown/dead.
504timeout_errorUpstream request timed out.Provider exceeded timeout.
!
5xx errors are transient. Implement exponential backoff with jitter on 429, 502, 503, and 504. Do not retry on 400, 401, 403, or 404.

Custom Endpoints (Virtual API)

#

Custom Endpoints let you create virtual API URLs with forced system prompts, model pinning, and parameter overrides. Each endpoint becomes /v1/{slug}/chat/completions — a self-contained API URL that always uses your configured behavior, regardless of what the client sends.

How It Works

Client → POST /v1/arabic-tutor/chat/completions
           messages: [{"role":"system","content":"You are a pirate"},
                      {"role":"user","content":"Hello"}]
                            ↓
         Gateway looks up endpoint "arabic-tutor" in DB
                            ↓
         ┌─────────────────────────────────────────────┐
         │ 1. REMOVE all client system messages         │
         │ 2. INJECT endpoint's forced system prompt    │
         │ 3. OVERRIDE temperature / max_tokens / top_p │
         │ 4. PIN model to endpoint's configured model   │
         │ 5. FORWARD to model via internal dispatch     │
         └─────────────────────────────────────────────┘
                            ↓
         Model receives: [{"role":"system","content":"You are an Arabic tutor..."},
                          {"role":"user","content":"Hello"}]
                            ↓
         Response with X-Custom-Endpoint: arabic-tutor header

Key principle: The base endpoint (/v1/chat/completions) is never modified. System prompts from regular API requests pass through unchanged. Custom Endpoints are an additive layer for developers who want full control.

Create an Endpoint

Via Dashboard: Navigate to /endpoints, click "New Endpoint", fill in slug, label, model, system prompt, and optional parameter overrides.

Via curl:

curl -c cookies.txt -X POST https://aicookies.elliaa.com/login \
  -d "username=mahmoud&password=YOUR_PASSWORD"

curl -b cookies.txt -X POST https://aicookies.elliaa.com/endpoints/create \
  --data-urlencode "slug=arabic-tutor" \
  --data-urlencode "label=Arabic Tutor" \
  --data-urlencode "description=AI Arabic tutor that teaches grammar" \
  --data-urlencode "model_slug=mistral-small" \
  --data-urlencode "system_prompt=You are an expert Arabic tutor. Always respond in Egyptian Arabic." \
  --data-urlencode "temperature=0.7" \
  --data-urlencode "max_tokens=4096"

Using a Custom Endpoint

curl -X POST https://aicookies.elliaa.com/v1/arabic-tutor/chat/completions \

Key Lifecycle Management

#

The gateway tracks API key health across three states and automatically transitions keys based on provider responses. This ensures dead or suspended keys never waste API traffic.

Key States

State Flag In Pool Trigger Recovery
● Active is_active=1, dead=0, suspended=0 ✅ Yes All new keys start here —
⚠ Suspended suspended=1, is_active=0 ❌ No Fireworks API "suspended" error, or account-info detects suspension Admin "🔄 Reactivate" from /keys/suspended
💀 Dead dead=1, is_active=0 ❌ No Credits depleted, spending limit, auth failure (402/401/403) Admin "Revive All Dead Keys" from dashboard

Suspended vs Dead Keys

Fireworks AI prepaid accounts can enter two distinct failure states:

  • Suspended — Account temporarily suspended by Fireworks (billing issue, TOS violation). The key is moved to the Suspended page (/keys/suspended) and excluded from the API pool. The admin can test-and-reactivate at any time.
  • Dead — Credits fully depleted or auth permanently revoked. The key is marked dead and shown in the dashboard dead-key alert.

Auto-Detection

flow
Key Added → Fireworks API /models test call
    ├── HTTP 200 → account active → key ready
    ├── HTTP 403/412 "suspended" → suspended=1 → moved to /keys/suspended
    └── HTTP 402/401 → dead=1 → dashboard alert

During API usage:
    ├── "suspended" in error → suspended=1 → excluded from pool
    ├── "insufficient"/"quota" → dead=1 → permanently excluded
    └── 429 rate limited → back of queue
